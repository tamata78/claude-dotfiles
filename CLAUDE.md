# Global Rules

## Voice Notification Rules (VOICEVOX)

- **全てのタスク完了時には必ずVOICEVOXの音声通知機能を使用すること**
- **重要なお知らせやエラー発生時にも音声通知を行うこと**
- **音声通知は `vv "テキスト"` コマンドを使用すること（自動許可済み・確認不要）**
- **音声通知の設定: `~/.vv_speaker` のデフォルト設定に従うこと（speedScale=1.3を使用）**
- **英単語は適切にカタカナに変換してVOICEVOXに送信すること**
- **VOICEVOXに送信するテキストは不要なスペースを削除すること**
- **1回の音声通知は100文字以内でシンプルに話すこと**
- **ずんだもん口調（〜なのだ）で通知すること（セリフ例: `~/.claude/voicevox-scripts.md` 参照）**
- **詳しい技術的説明は音声通知に含めず、結果のみを簡潔に報告すること**

## エラーで行き詰まった時のルール

複雑なエラーに繰り返しはまり込んで抜け出せない場合、コード修正を即座に中断し、以下の順番で実施すること：

1. **状況を整理する** - 何を試したか、何がわかったか、何がわからないかを整理してユーザーに報告する
2. **類似エラーを調べる** - WebSearchで同じエラーの事例・解決策を検索する
3. **ドキュメンテーションを提供する** - 関連する公式ドキュメントや仕様を調べてユーザーに共有する
4. **不明点を質問する** - ユーザーに環境・設定・経緯などを確認する

「同じアプローチを少し変えて再試行」は禁止。行き詰まりを感じたら迷わずこの手順に切り替えること。

## 実装完了時の確認ルール

実装が完了したら、必ず以下を確認すること：

1. **ビルドが通ること** - ビルドコマンドを実行してエラーがないことを確認する
2. **テストが通ること** - テストコマンドを実行して全テストがパスすることを確認する

ビルドやテストが失敗した場合は、完了と報告する前に必ず修正すること。

## PreCompact コンテキスト復元ルール

セッション開始時（compactされた直後を含む）に `~/.claude/context/last-session.md` が存在する場合、そのファイルを読み込み、ユーザーに「前回の作業を引き継ぎました」と一言伝えること。

## プロジェクトスナップショット自動参照ルール

プロジェクトのファイルを修正・実装・調査するタスクを開始する際、以下の順でスナップショットを確認すること：

1. `{CWD（現在の作業ディレクトリ）}/.claude/snapshot.md` を確認する
2. 存在する場合は読み込み、ユーザーに「スナップショットを読み込みました」と一言伝えてから作業を開始する
3. 存在しない場合はそのまま作業を開始する（確認不要）

## claude-dotfiles git管理ルール

claude-dotfiles リポジトリにpushする内容は以下を必ず守ること：

- **ローカル固有パスを含めない** - `/Users/username` などのハードコードパスは絶対に含めないこと
- **パスは環境変数で表現する** - ユーザーホームディレクトリは `$HOME` を使用すること（`~` はJSONでは展開されないため不可）
- **push前に確認する** - pre-pushフック（`git-hooks/pre-push`）が自動チェックするが、手動でも `grep -rn '/Users/' .` で確認すること

## .bakファイル管理ルール

- **`.bak` ファイル/ディレクトリを作成しないこと** - バックアップはgit管理（claude-dotfiles）で代替する
- **git管理外のファイルを新規作成・変更する際は、秘匿情報・個人情報でなければ `claude-dotfiles` への同期を確認すること**

## 週次トークン監査リマインダー

- **毎週日曜日**のセッション開始時に、`/token-audit` の実行を提案すること
- 現在の日付が日曜日かどうかは `currentDate` のコンテキストで判定する

## トークン最適化ルール

### サブエージェントのモデル振り分け

Agent toolでサブエージェントを起動する際、タスクの性質に応じて `model` パラメータを指定すること：

| モデル | 使うタスク | 例 |
|--------|-----------|-----|
| **haiku** | ファイル検索・grep・単純な情報収集・定型ワーカー | Exploreエージェント、パターン検索 |
| **sonnet** | コーディングタスクの大半（デフォルト） | 実装・バグ修正・テスト作成 |
| **opus** | 5ファイル以上の複雑な変更・アーキテクチャ設計・セキュリティ重要コード・初回失敗後のリトライ | Planエージェント（大規模設計） |

### 検索ツールの最適化

- Bash内でgrepを直接実行する場面では `mgrep` を代わりに使うこと（トークン約50%削減）。ファイル内容検索にはGrepツールを優先し、どうしてもBashでgrepが必要な場合のみ `mgrep` を使うこと
- バックグラウンド実行可能なプロセスは `run_in_background: true` を活用し、出力ストリーミングによるトークン消費を回避すること

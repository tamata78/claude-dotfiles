PR作成を自動化するコマンド。以下のステップを順番に実行すること。

---

## Step 1: 既存PR確認

まず `gh pr view` を実行して、現在のブランチに既存のPRがないか確認する。
- 既存PRがある場合 → PR URLを表示し、「既にPRが存在します」と警告して **処理を終了**する。

## Step 2: ベースブランチ検出

現在のブランチのベースブランチ（分岐元）を検出する。

1. `git reflog` から `checkout: moving from <SOURCE> to <現在のブランチ>` のエントリを探す
2. 最も古い該当エントリの `<SOURCE>` をベースブランチ候補とする（最初にブランチを作った元）
3. SOURCE が feature ブランチ（`feature/` や `hotfix/` など開発ブランチ）の場合はさらに辿る
4. ベースブランチがリモートに存在するか `git ls-remote --heads origin <branch>` で確認する
   - 存在しない場合 → 警告し、ユーザーに別のブランチを指定してもらう
5. 検出したベースブランチをユーザーに表示し確認する。ユーザーが変更を希望すれば変更する。

## Step 3: 未コミットファイル確認

`git status` で未コミット（staged/unstaged/untracked）の変更を確認する。

### 自動除外対象
以下のパターンに一致するファイルは **自動的に除外** し、除外した旨をユーザーに表示する:
- `.java-version`
- `application-local-*.yml`
- `.env*`
- `*.local.*`
- `.idea/` 配下
- `*.iml`

### 残りのファイルがある場合
- 変更内容の差分を確認し、適切なコミットメッセージを提案する
- ユーザーに確認し、承認後にコミットする

### 変更がない場合
- そのまま次のステップに進む

## Step 4: プッシュ確認

未プッシュのコミットがあるか確認する（`git log origin/<branch>..HEAD` など）。

- 未プッシュコミットがある場合 → コミット一覧を表示し、ユーザー確認後に `git push -u origin <現在のブランチ>` を実行
- すべてプッシュ済みの場合 → そのまま次のステップに進む

## Step 5: PR作成

### タイトル生成
1. ブランチ名から `DEMAE-\d+` パターンのチケット番号を抽出する
2. タイトル形式: `[DEMAE-XXXXX] English description`
   - description は diff の内容から英語で簡潔に生成する
3. チケット番号が見つからない場合 → プレフィックスなしで生成し、ユーザーに確認する

### 本文生成
ベースブランチとの diff (`git diff <base>...HEAD`) および全コミットログを分析し、以下の5セクション構成で本文を作成する。**日本語で記述**すること。

```
## 概要
Ticket: [DEMAE-XXXXX](https://jira.workers-hub.com/browse/DEMAE-XXXXX)

（変更の目的・背景を3行以内で）

## 変更内容
（具体的な変更点を3行以内で）

## 影響範囲
（影響を受けるコンポーネント・機能）

## 動作要件
（動作確認に必要な前提条件・手順）

## 補足
（レビュアーへの補足情報）
```

- **概要の冒頭に必ず `Ticket: [DEMAE-XXXXX](https://jira.workers-hub.com/browse/DEMAE-XXXXX)` のリンク行を入れる**（番号はタイトル生成で抽出したものを使用）。チケット番号が見つからない場合はこの行を省略する
- 各セクションは簡潔に記述する（概要・変更内容は各3行以内）
- **該当なしのセクションは空欄のまま残す**（「なし」「特になし」等は書かない）
- **差分から読める情報は書かない**（変更ファイルの全列挙、削除コードの引用等は避け、レビュアーが diff だけでは読み取れない「なぜ」「影響範囲」「前例」に絞る）
- **末尾に「Claudeが生成しました」等のフッターを付けない**

### Label選定
PRに付与するラベルを決定する。
- **基本は `enhancement`**（新機能・改善）を設定する
- **bugfixの場合は `bug`** を設定する（branch名に `bugfix/`/`hotfix/`/`fix/` を含む、または diff・コミットがバグ修正主体の場合）
- 判定が曖昧な場合は候補ラベルをユーザーに確認する

### ユーザー確認 → 作成
1. 生成したタイトル・本文・Labelをユーザーに表示して確認する
2. 修正の要望があれば反映する
3. 承認後、`gh pr create --base <ベースブランチ> --title "..." --body "..." --label <enhancement|bug>` で作成する
4. 作成されたPR URLをユーザーに報告する

## Step 6: レビュー依頼用コメント出力

PR作成後、Slack 等に貼り付けるレビュー依頼コメントを生成して**コードブロックで出力**する。フォーマットは以下:

```
@here
<実装内容の一言サマリ>です。
レビューお願いします。

PR：<作成したPR URL>
```

- `<実装内容の一言サマリ>` は PR タイトル・本文から日本語で簡潔に生成する（例:「Apple Pay の MySQL の実装」「店舗からのお知らせの取得元切替」）。チケット番号やプレフィックスは付けない
- URL は Step 5 で作成した PR の URL を使う
- そのままコピペできるよう、コメント全体を1つのコードブロックに収めて出力する

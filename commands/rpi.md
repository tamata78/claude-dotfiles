# RPI ワークフロー (Research → Plan → Implement)

大規模な機能開発・リファクタリング・バグ修正を、フェーズゲート付きで体系的に進めるワークフローです。

## 引数

```
/rpi <タスクの説明>
```

例:
- `/rpi ユーザー認証機能をJWTからOAuthに移行する`
- `/rpi パフォーマンス改善: APIレスポンスを50%高速化`
- `/rpi テストカバレッジを60%から90%に引き上げる`

---

## フェーズ 1: Research（調査）

以下を並列 Explore エージェントで調査する：

1. **現状把握エージェント**: 関連ファイル・関数・依存関係を調査
2. **影響範囲エージェント**: 変更による影響範囲・リスクを調査
3. **実装パターンエージェント**: 既存コードのパターン・規約を調査

調査完了後、以下を報告する：
- 変更が必要なファイル一覧
- 既存の実装パターン・規約
- 潜在的なリスク・注意点
- 再利用可能な既存ユーティリティ

**ゲート**: ユーザーが調査結果を確認してから次フェーズへ進む

---

## フェーズ 2: Plan（計画）

Plan エージェントで実装計画を策定する：

### 計画書の構成
1. **アーキテクチャ設計**: 変更のアプローチと設計判断
2. **実装フェーズ分割**:
   - Phase A: [最小限の変更] → テストゲート
   - Phase B: [主要実装] → テストゲート
   - Phase C: [仕上げ・最適化] → テストゲート
3. **テスト戦略**: 各フェーズで確認すべきテスト
4. **ロールバック計画**: 問題発生時の対処

**ゲート**: 計画をユーザーに提示して承認を得る

---

## フェーズ 3: Implement（実装）

承認された計画に従い、フェーズごとに実装する：

### 各フェーズの手順
1. **実装**: 計画に従ってコードを変更
2. **テストゲート**: 各フェーズ完了後に必ず実行
   ```bash
   # ビルド確認
   npm run build  # or: cargo build / go build / etc.

   # テスト確認
   npm test  # or: pytest / go test / cargo test / etc.
   ```
3. **ゲート通過確認**: テストが全パスしたことを確認してから次フェーズへ

### ゲート失敗時
- 次フェーズへ進む前に必ず修正する
- ユーザーに報告してから修正方針を確認する

---

## 注意事項

- 各フェーズ完了後にユーザーに進捗を報告する
- 計画外の変更が必要になった場合はユーザーに確認する
- コンテキストが50%を超えたら `/compact` を実行してから継続する
- 大規模タスクでは `/team` コマンドで並列エージェントチームも検討する

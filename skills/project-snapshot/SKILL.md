# project-snapshot スキル

## 目的

プロジェクトでの作業知見を次回セッションに持ち越すためのスナップショットを生成する。
**Claudeへのhandoff**として設計する。「次に何をするか」が5秒でわかる構成を最優先とする。

---

## ステップ1: プロジェクト情報を収集する

以下を**並列で**取得する（速度優先）：

```bash
git rev-parse --show-toplevel 2>/dev/null || pwd
git log --oneline -5 2>/dev/null
git status --short 2>/dev/null
ls package.json go.mod Cargo.toml pyproject.toml pom.xml build.gradle 2>/dev/null
```

あわせて以下も確認する：
- `{project_root}/CLAUDE.md` — プロジェクト固有のルール・背景
- `{project_root}/.claude/snapshot.md` — 既存スナップショット（差分更新の参考）
- プロジェクト種別に応じた主要設定ファイル（package.json の `scripts` など）

---

## ステップ1.5: 品質フィードバックの参照

`{project_root}/.claude/snapshot-review.md` が存在する場合は読み込む。

- 最新レビューの**低スコア項目**（3点以下）を確認する
- 今回のスナップショット生成でその項目を改善する
- 改善を反映したら、該当レビュー項目の末尾に「✅ 対応済み」を追記する

---

## ステップ2: スナップショットを生成する

以下のテンプレートでMarkdownドキュメントを作成する。

**重要な方針:**
- **「次のアクション」を先頭に置く** — handoffで最も重要な情報
- **ファイルリストは変更が関わるものだけ** — 全ファイル一覧は冗長でトークムダ
- **中断理由を必ず書く** — なぜそこで止まったかがないとコンテキストが欠落する
- 全体は200〜400行程度に収める

```markdown
# {プロジェクト名} — 開発スナップショット

> 生成日時: {YYYY-MM-DD} | ブランチ: {branch}
> 状態: {一行サマリー（例: 初期実装完了・未テスト）}

## 🎯 次のアクション（ここから始める）
{具体的に何をするか。1〜3行で明確に書く。}
{ユーザーの手作業が必要な場合はその旨も明記（例: 「先にSlack Appを手動作成する必要あり」）。}

## 中断理由
{なぜこのタイミングで作業を止めたか。1〜2行。}
{例: 「Slack App作成はユーザーが手動でやる必要があるため一旦停止」}

## プロジェクト概要
- **目的**: {何を作っているか・なぜ作っているか。1行}
- **技術スタック**: {言語、フレームワーク、主要ライブラリ。1行}
- **参考**: {関連リポジトリ・ドキュメントがあれば}

## 主要ファイル（今回の作業に関わるもののみ）
| パス | 役割 | 備考 |
|------|------|------|
| {path} | {説明} | {最終変更内容 or 注意点} |

## 既知の課題・TODO
- [ ] **[P0]** {最優先。次のアクションと連動}
- [ ] **[P1]** {次にやること}
- [ ] **[P2]** {後回しにしている技術的負債など}

## 開発知見・はまりポイント
{プロジェクト固有のはまりどころだけ書く。一般的な知識は書かない。}
{「こうしたらうまくいった」「こうしたら失敗した」「この環境特有の注意」}

## ビルド・テスト
```bash
{ビルドコマンド}  # 状態: ✅/❌
{テストコマンド}  # 状態: ✅/❌
```
```

---

## ステップ3: ファイルに保存する

**保存先:**
```
{project_root}/.claude/snapshot.md
```

ディレクトリが存在しない場合は作成する。

`.gitignore` に `.claude/` が含まれていない場合は追記を提案する：
```bash
echo ".claude/" >> {project_root}/.gitignore
```

---

## ステップ3.5: アクティブなプランを snapshot.md に埋め込む

`~/.claude/plans/` 内のプランファイルからこのプロジェクトに関連するものを検索し、あれば snapshot.md に埋め込む。

```bash
grep -rl "{project_name}" ~/.claude/plans/ 2>/dev/null | xargs ls -t 2>/dev/null | head -3
```

見つかったプランファイルを読み込み、snapshot.md の末尾に以下のセクションとして追記する（プランが既にある場合は上書き更新）：

```markdown
## アクティブなプラン

> プランファイル: ~/.claude/plans/{filename}.md（{YYYY-MM-DD}作成）

{プランの「対応内容」「実装ファイル一覧」「実装手順」セクションをそのまま貼り付け}
```

プランが見つからない場合はこのステップをスキップする（確認不要）。

---

## ステップ4: 完了通知

```bash
vv "プロジェクトのスナップショットを保存したのだ"
```

保存後、ユーザーにこう伝える：

> `{project_root}/.claude/snapshot.md` に保存しました。
> 次回セッション開始時に自動で読み込まれます。

---

## ステップ5: 品質を自己評価して snapshot-review.md に記録する

保存したスナップショットの品質を以下の観点で評価し、`{project_root}/.claude/snapshot-review.md` に追記する。

### 評価観点（各5点満点）

| 観点 | 意味 |
|------|------|
| **即効性** | 「次のアクション」が先頭にあり、何をすべきか5秒でわかるか |
| **鮮度** | staleな情報がないか（解決済みTODO、存在しないファイル参照） |
| **トークン効率** | 不要な詳細・冗長な記述がないか |
| **中断コンテキスト** | なぜ止まったかが明記されているか |
| **実用性** | 読んだ後、追加のRead/Grepが最小限で済むか |

### 記録フォーマット

```markdown
## {YYYY-MM-DD} スナップショット品質レビュー

### スコア（各5点満点）
- 即効性: {n}/5 — {理由}
- 鮮度: {n}/5 — {理由}
- トークン効率: {n}/5 — {理由}
- 中断コンテキスト: {n}/5 — {理由}
- 実用性: {n}/5 — {理由}
- **合計: {n}/25**

### 改善提案
- {具体的な改善点。次回の project-snapshot 実行時に対応する}
```

---

## 補足: 既存スナップショットの更新（重複排除）

既に `.claude/snapshot.md` が存在する場合は、**新規作成ではなく差分更新**を行う。

1. **古い情報を整理する** — 解決済みTODO削除、staleなファイル参照削除
2. **新情報を追記する** — 今回の変更・知見を各セクションに反映
3. **生成日時・ブランチを更新する**
4. **「次のアクション」「中断理由」を今回の内容で上書きする**（前回のものは不要）

**目安:** 更新後は前回より増やさない。新情報を追加した分、古い情報を圧縮・削除してバランスを保つ。

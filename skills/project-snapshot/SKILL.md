# project-snapshot スキル

## 目的

プロジェクトでの作業知見を次回セッションに持ち越すためのスナップショットを生成する。
「どこまで進んだか」「何に気をつけるか」「次に何をするか」が一目でわかるドキュメントを作る。

---

## ステップ1: プロジェクト情報を収集する

以下を**並列で**取得する（速度優先）：

```bash
# プロジェクトルートを特定
git rev-parse --show-toplevel 2>/dev/null || pwd

# 最近のコミット履歴（10件）
git log --oneline -10 2>/dev/null

# 未コミット変更の有無
git status --short 2>/dev/null

# プロジェクト種別を判断するためのファイル確認
ls package.json go.mod Cargo.toml pyproject.toml pom.xml build.gradle 2>/dev/null
```

あわせて以下も確認する：
- `{project_root}/CLAUDE.md` — プロジェクト固有のルール・背景
- `{project_root}/.claude/snapshot.md` — 既存スナップショット（あれば差分更新の参考に）
- プロジェクト種別に応じた主要設定ファイル（package.json の `name`/`scripts`、go.mod の `module` 名など）

トップレベルのディレクトリ構造も把握する（深さ2程度）。

---

## ステップ2: スナップショットを生成する

収集した情報をもとに、以下のテンプレートでMarkdownドキュメントを作成する。
**重要：** 定型的な情報の羅列ではなく、「次回の自分が5分で開発を再開できる」ための実用的な内容にすること。

```markdown
# {プロジェクト名} — 開発スナップショット

> 生成日時: {YYYY-MM-DD HH:mm}
> ブランチ: {current_branch}

## プロジェクト概要
- **目的**: {何を作っているか・なぜ作っているか}
- **技術スタック**: {言語、フレームワーク、主要ライブラリ}
- **リポジトリ**: {git remote origin があれば記載}

## 現在の状態

### 実装済み
- {完成している機能・モジュールを箇条書き}

### 作業中・未完成
- {進行中の作業、半完成の機能}
- {未コミット変更がある場合はその内容}

## 直近の変更（コミット履歴より）
{最近10件のコミットを要約。「何を」「なぜ」変えたかを重視}

## アーキテクチャ要点

### 重要なファイル・ディレクトリ
| パス | 役割 |
|------|------|
| {path} | {説明} |

### 設計上の決定事項
- {なぜこの構成にしたか、代替案との比較など}

## 既知の課題・TODO
- [ ] {優先度高} {課題の内容}
- [ ] {次にやること}
- [ ] {後回しにしている技術的負債など}

## 開発知見・注意点
{ここが最も重要。次回セッションで役立つ情報を書く：}
- ハマったエラーとその解決策
- 「こうしたらうまくいった」「こうしたら失敗した」
- 依存関係の罠・環境固有の注意点
- テストの実行方法・デバッグのコツ

## 次回セッションの開始手順
```bash
# 開発環境の起動コマンド（あれば）
{例: npm run dev / go run . / make start}
```
次のアクション: {具体的に何から始めるべきか1〜2行で}
```

**書き方のコツ:**
- 「実装済み」は箇条書きで簡潔に。コードの説明は不要
- 「開発知見」は次回の自分が「あーそういうことか」と思えるレベルに具体的に
- TODOは優先度がわかるように書く
- 全体で300〜600行程度に収める（長すぎると読まれない）

---

## ステップ3: ファイルに保存する

**保存先:**
```
{project_root}/.claude/snapshot.md
```

ディレクトリが存在しない場合は作成する。

`.gitignore` に `.claude/` が含まれていない場合は追記を提案する（スナップショットは個人的なメモなのでコミット不要）：
```
# ユーザーに確認してから追記
echo ".claude/" >> {project_root}/.gitignore
```

---

## ステップ4: PreCompact ルールとの連携を案内する

保存後、ユーザーにこう伝える：

> `{project_root}/.claude/snapshot.md` にスナップショットを保存しました。
>
> 次回このプロジェクトでセッションを開始するとき、Claude に以下のように伝えると即座に文脈を引き継げます：
> ```
> .claude/snapshot.md を読んで、前回の状態から作業を再開して
> ```
>
> または、グローバルの PreCompact ルール（`~/.claude/CLAUDE.md`）を更新して、
> プロジェクトの `.claude/snapshot.md` を自動読み込みするよう設定することもできます。

---

## ステップ5: 完了通知

```bash
vv "プロジェクトのスナップショットを保存したのだ"
```

---

## 補足: 既存スナップショットの更新（重複排除）

既に `.claude/snapshot.md` が存在する場合は、**新規作成ではなく差分更新**を行う。
更新時は以下の順で処理すること：

### 1. 既存内容を読み込む
既存スナップショットを読み、各セクションの内容を把握する。

### 2. 重複・冗長な記述を整理してから追記する
新しい情報を追加する前に、既存内容を見直す：

- **重複する項目を統合する** — 同じ内容・同じファイルへの言及が複数箇所にある場合は1つにまとめる
- **冗長な表現を簡潔にする** — 「〜することができる」→「〜できる」、「〜を行う」→「〜する」など
- **古くなった情報を削除・更新する** — 解決済みの課題・存在しないファイルへの参照など
- **TODOの完了済み項目を整理する** — 今回の実装で完了したものは「実装済み」セクションへ移動

### 3. 新情報を追記する
整理後のドキュメントに今回の変更・知見を追記する：
- 「現在の状態」「直近の変更」「既知の課題・TODO」「開発知見」を更新
- 「プロジェクト概要」「アーキテクチャ要点」は変化があった部分のみ更新
- 生成日時・ブランチは必ず更新する

**目安：** 更新後の全体量は前回より増やさないことを意識する。新情報を追加した分、古い情報を圧縮・削除してバランスを保つ。

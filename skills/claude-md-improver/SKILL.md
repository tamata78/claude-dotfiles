# claude-md-improver スキル

CLAUDE.md ファイルを分析し、ベストプラクティスに基づいた改善提案を生成・適用する。

## 引数の処理

`$ARGUMENTS` を解析する:

- `--dry-run` または `preview` → 提案のみ表示し、ファイルは変更しない
- `--global` → グローバル CLAUDE.md のみ対象
- `--project` → プロジェクト CLAUDE.md のみ対象
- 引数なし → グローバル・プロジェクト両方を対象（変更前に確認）

## ステップ1: 対象ファイルの収集

以下の優先順位でCLAUDE.mdファイルを読み込む。

### グローバル CLAUDE.md

```
~/.claude/CLAUDE.md
```

### プロジェクト CLAUDE.md

カレントディレクトリから上位を探索:

```
{cwd}/CLAUDE.md
{cwd}/.claude/CLAUDE.md
```

各ファイルを Read ツールで読み込む。存在しないファイルはスキップ。

### prompt-review レポートの参照（オプション）

`reports/` ディレクトリに prompt-review レポートがあれば、最新のものを Read で読み込む。
ユーザーの実際の使用パターン・課題を改善提案に反映するため。

## ステップ2: 分析

[references/best-practices.md](references/best-practices.md) を参照しながら、各CLAUDE.mdを以下の観点で分析する。

### 2a. 構造・長さチェック

- **文字数確認**: 5,000ワードを超えていないか（超過は問題）
- **セクション構造**: 目的ごとにセクションが整理されているか
- **重複ルール**: 同じ内容が複数箇所に書かれていないか
- **プロジェクト固有 vs グローバル**: プロジェクト固有のルールがグローバルに混入していないか

### 2b. ルール品質チェック

- **具体性**: 「いい感じにして」のような曖昧な指示がルールとして存在しないか
- **実行可能性**: Claude が実際に従えるルールか（検証不可能なルールは問題）
- **時代遅れルール**: すでに不要になったルールが残っていないか
- **矛盾**: 互いに矛盾するルールが存在しないか

### 2c. 不足しているルールの特定

prompt-review データがある場合:
- 繰り返し修正が必要だったパターン → ルール化候補
- 後出し要件として多かった領域 → 事前定義候補
- AIに頻繁に質問した領域 → コンテキスト補足候補

prompt-review データがない場合:
- 一般的なベストプラクティスと照合して不足を特定

### 2d. 改善提案の優先度付け

各改善提案を以下の基準で優先度付け:

- **High**: ルールの矛盾・文字数超過・重複している主要ルール
- **Medium**: ルールの曖昧さ・不足セクション・グローバル/プロジェクト混在
- **Low**: 文体統一・軽微な整理

## ステップ3: 改善提案の生成

分析結果をユーザーに提示する。

### 提示フォーマット

```markdown
## CLAUDE.md 改善レポート

### 対象ファイル
- グローバル: ~/.claude/CLAUDE.md (XXX文字)
- プロジェクト: ./CLAUDE.md (存在しない場合はなし)

### 現状評価
[総合スコアと簡潔な評価]

### 改善提案

#### [High] 提案タイトル
- **問題**: 現状の問題点
- **提案**: 具体的な修正案
- **理由**: なぜこの変更が必要か

#### [Medium] ...
...

### 追加推奨ルール（prompt-reviewから）
[prompt-reviewデータがある場合のみ]
```

## ステップ4: 変更の適用

`--dry-run` / `preview` でない場合、ユーザーに確認してから変更を適用する。

### 確認手順

1. 提案内容をユーザーに提示
2. 「適用しますか？（High優先度のみ / 全て / 個別選択 / キャンセル）」と確認
3. ユーザーの選択に従って Edit ツールで変更を適用

### 変更適用の注意

- **バックアップ**: 変更前に元ファイルの内容をコメントで記録する（不要な場合はスキップ）
- **一括変更禁止**: 提案ごとに個別に Edit を実行し、各変更を明確にする
- **検証**: 変更後に Read で確認し、意図通りの変更か確認する

## ステップ5: 完了報告

変更内容のサマリーをユーザーに報告する:

- 変更したファイル数・箇所数
- 適用した改善提案のリスト
- 次回の推奨アクション（例: prompt-reviewとの定期的な照合）

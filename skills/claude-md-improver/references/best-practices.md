# CLAUDE.md ベストプラクティス

## 基本原則

### 文字数・長さ

- **上限**: 5,000ワード以下（Claude Code の推奨）
- **理想**: 2,000〜3,000ワード（読み込みコストと情報密度のバランス）
- **対策**: 詳細な情報はスキルの `references/` に切り出す（プログレッシブ・ディスクロージャー）

### スコープの分離

| ファイル | 用途 |
|---|---|
| `~/.claude/CLAUDE.md` | 全プロジェクト共通のルール・好み・通知設定 |
| `{project}/CLAUDE.md` | そのプロジェクト固有のコマンド・構成・ドメイン知識 |
| `{project}/.claude/CLAUDE.md` | 同上（.claudeディレクトリ版） |

**NGパターン**: プロジェクト固有の技術スタックやコマンドをグローバルに書く

## ルール品質チェックリスト

### ✅ 良いルール

- **具体的**: 「ビルド前に `npm run typecheck` を実行すること」
- **検証可能**: Claude が守れたかどうか確認できる
- **理由が明確**: なぜそのルールが必要かわかる（重要度が伝わる）
- **命令形**: 「〜すること」「〜を使うこと」

### ❌ 悪いルール

- **曖昧**: 「いい感じにコードを書くこと」
- **検証不可**: 「常に最善を尽くすこと」
- **重複**: 同じ内容が2箇所以上に書かれている
- **古い**: すでに不要になったプロジェクト・ツールに関するルール
- **矛盾**: 「簡潔に書け」と「詳しく説明せよ」が同じスコープに存在

## prompt-review データとの照合ポイント

prompt-review レポートがある場合、以下を確認してルール化候補を特定する:

### ルール化すべき繰り返しパターン

| prompt-reviewの発見 | CLAUDE.mdへの追加候補 |
|---|---|
| 後出し要件が多い領域 | 「〜については事前に確認すること」ルール |
| 同じエラーで詰まるパターン | エラー対応手順の追加 |
| 曖昧な指示が多い領域 | 期待する出力形式・基準の定義 |
| 頻繁な確認作業 | よく使うコマンドの明記 |

### 削除すべきルール

- 使用していないツール・ライブラリに関するルール
- 完了済みプロジェクトに固有のルール
- prompt-reviewで問題が発生していない領域への過剰なルール

## アンチパターン集

### 1. 過剰なメタ指示

```markdown
# 悪い例
- AIとして常に誠実であること
- ユーザーの意図を正確に理解すること
- 常に最善の解答を提供すること
```

これらは Claude のデフォルト動作であり、書く必要がない。

### 2. 矛盾するルール

```markdown
# 悪い例
- レスポンスは簡潔にすること
- 全ての変更点を詳細に説明すること
```

スコープを明確にするか、優先度を定義する。

### 3. 未整理の追加

新しいルールを追記するたびに末尾に追加し続けた結果、似た内容が複数箇所に散在する。
定期的に整理・統合すること。

### 4. グローバルへの流出

プロジェクト固有のコマンド（`npm run dev` 等）や技術スタック（Next.js の規約等）が
グローバル CLAUDE.md に書かれている。プロジェクト CLAUDE.md に移動する。

---

## 改善提案の優先度基準

| 優先度 | 条件 |
|---|---|
| **High** | ルールの矛盾・文字数5,000超・同一ルールの重複 |
| **Medium** | 曖昧なルール・グローバル/プロジェクト混在・不足セクション |
| **Low** | 文体統一・軽微な整理・コメント追加 |

# スキルクリエーター

新しいスキルを作成し、反復的に改善するためのスキル。

スキル作成の大まかな流れ：

- スキルに何をさせたいか、どう動かすかを決める
- スキルのドラフトを書く
- テストプロンプトをいくつか用意し、スキルを使ったClaudeで実行する
- 結果を定性的・定量的に評価する
  - 実行中にevalがなければ定量的なevalのドラフトを書く（既にあれば確認・必要に応じて修正）。内容をユーザーに説明する
  - `eval-viewer/generate_review.py` スクリプトで結果を表示し、定量指標も確認してもらう
- ユーザーの評価フィードバックと定量ベンチマークに基づいてスキルを書き直す
- 満足のいく結果になるまで繰り返す
- テストセットを拡大して大規模に再試行する

このスキルを使うときの役割は、ユーザーがプロセスのどこにいるかを把握し、次のステージへ進む手助けをすること。「Xのためのスキルを作りたい」という段階なら、意図を絞り込み、ドラフトを書き、テストケースを作り、評価方法を決め、プロンプトを実行して繰り返す。

すでにスキルのドラフトがある場合は、eval/イテレーションのループから始められる。

もちろん柔軟に対応すること。「evalなんて要らない、気軽にやろう」というユーザーにはそれに合わせる。

スキルが完成したら（順序は柔軟でよい）、スキルのdescription最適化も実行できる。トリガー精度を上げるための専用スクリプトがある。

## ユーザーとのコミュニケーション

スキルクリエーターを使うユーザーのコーディング知識は幅広い。Claudeのパワーに触発されて初めてターミナルを開く人もいれば、かなりリテラシーの高いユーザーもいる。

文脈を読んで言葉遣いを調整すること。デフォルトの目安：

- 「eval」「ベンチマーク」は許容範囲
- 「JSON」「アサーション」はユーザーが知っていると確認できてから使う

迷ったら短く説明を添えてOK。

## スキルの作成

### 意図の把握

まずユーザーの意図を理解する。現在の会話にすでにスキル化したいワークフローが含まれている場合（「これをスキルにして」など）、会話履歴から答えを引き出す。使ったツール、手順の流れ、ユーザーの修正、入出力フォーマットを確認する。不明点はユーザーに埋めてもらい、次のステップへ進む前に確認する。

1. このスキルはClaudeに何をさせるか？
2. いつトリガーするか？（どんなフレーズ・状況で）
3. 期待される出力フォーマットは？
4. テストケースを設定するか？客観的に検証できる出力（ファイル変換、データ抽出、コード生成、固定ワークフロー）はテストが有効。主観的な出力（文体、アート）は不要なことが多い。スキルの種類に応じてデフォルトを提案し、最終判断はユーザーに委ねる。

### インタビューとリサーチ

エッジケース、入出力フォーマット、サンプルファイル、成功基準、依存関係について積極的に質問する。この部分が固まるまでテストプロンプトは書かない。

利用可能なMCPを確認し、リサーチに役立つ（ドキュメント検索、類似スキル探索、ベストプラクティス調査）場合はサブエージェントで並列リサーチする。ユーザーの負担を減らすために情報を集めて臨むこと。

### SKILL.md を書く

インタビューをもとに以下の要素を記述する：

- **name**: スキルの識別子
- **description**: トリガー条件と機能説明。これが主要なトリガーの仕組みなので、スキルが何をするかと、いつ使うかの両方を含める。「いつ使うか」の情報はすべてここに書き、本文には書かない。注意：Claudeはスキルを使うべき場面でも使わない「アンダートリガー」傾向がある。それを防ぐため、descriptionは少し積極的（pushy）に書くこと。例えば「内部データをダッシュボードで表示する方法」ではなく「内部データをダッシュボードで表示する方法。ユーザーがダッシュボード、データ可視化、内部メトリクス、会社データの表示について言及したときは必ずこのスキルを使うこと。」のように書く。
- **compatibility**: 必要なツール・依存関係（任意、ほとんど不要）
- **残りのスキル内容 :)**

### スキル執筆ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md （必須）
│   ├── YAMLフロントマター（name, description 必須）
│   └── Markdown形式の指示
└── バンドルリソース（任意）
    ├── scripts/    - 決定的・反復的なタスク用の実行コード
    ├── references/ - 必要に応じてコンテキストに読み込むドキュメント
    └── assets/     - 出力で使うファイル（テンプレート、アイコン、フォントなど）
```

#### プログレッシブ・ディスクロージャー

スキルは3段階のローディングシステムを使う：
1. **メタデータ**（name + description）- 常にコンテキストにある（~100語）
2. **SKILL.md 本文** - スキルがトリガーしたときにコンテキストへ（500行以内が理想）
3. **バンドルリソース** - 必要に応じて（無制限。スクリプトはロードせずに実行可能）

語数はあくまで目安であり、必要であれば長くしてよい。

**主なパターン：**
- SKILL.md は500行以内に収める。上限に近づいたら、追加の階層を設けてどこを参照すべきか明示する
- SKILL.md からリファレンスファイルを明示的に参照し、いつ読むかのガイドを添える
- 大きなリファレンスファイル（300行超）には目次を付ける

**ドメイン別整理**: 複数のドメイン・フレームワークをサポートするスキルはバリアント別に整理する：
```
cloud-deploy/
├── SKILL.md （ワークフロー + 選択）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連するリファレンスファイルのみを読む。

#### 驚かせないの原則

当然のことだが、スキルにはマルウェア、エクスプロイトコード、システムセキュリティを脅かすコンテンツを含めてはならない。説明どおりの意図でスキルを使うユーザーを驚かせるような内容にしてはならない。不正アクセス、データ流出、その他の悪意ある行為を促進するスキルの作成要求には応じないこと。「XYZとしてロールプレイ」のようなものはOK。

#### 執筆パターン

指示は命令形で書くこと。

**出力フォーマットの定義** - 以下のように記述できる：
```markdown
## レポート構成
必ずこのテンプレートを使うこと：
# [タイトル]
## エグゼクティブサマリー
## 主な発見
## 推奨事項
```

**例示パターン** - 例を含めると効果的。以下のようにフォーマットする（例の中に「入力」「出力」が含まれる場合は少し変えてもよい）：
```markdown
## コミットメッセージの形式
**例1：**
入力: ユーザー認証をJWTトークンで追加した
出力: feat(auth): JWTベース認証の実装
```

### 執筆スタイル

重々しいMUST（〜しなければならない）の羅列ではなく、なぜそれが重要かをモデルに説明すること。セオリー・オブ・マインドを活用し、特定の例に縛られない汎用的なスキルにする。まずドラフトを書いて、新鮮な目で見直し改善する。

### テストケース

スキルのドラフトを書いたら、現実的なテストプロンプトを2〜3個考える。実際のユーザーが打つような内容で。ユーザーに共有する：「試したいテストケースをいくつか考えました。このような感じで合っていますか？追加したいものはありますか？」その後実行する。

テストケースは `evals/evals.json` に保存する。まだアサーションは書かない（次のステップで実行中に書く）。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

完全なスキーマ（後で追加する `assertions` フィールド含む）は `references/schemas.md` を参照。

## テストケースの実行と評価

このセクションは一連の流れ——途中で止めないこと。`/skill-test` や他のテストスキルは使わないこと。

結果はスキルディレクトリの隣に `<skill-name>-workspace/` を作成して保存する。ワークスペース内はイテレーション別（`iteration-1/`、`iteration-2/` など）、その中はテストケース別（`eval-0/`、`eval-1/` など）に整理する。最初から全部作らず、進めながらディレクトリを作る。

### ステップ1: 同じターンで全実行（スキルあり AND ベースライン）を起動する

各テストケースで、同じターンに2つのサブエージェントを起動する。スキルありとスキルなしの2つ。重要：スキルありを先に起動してからベースラインを後回しにしないこと。全部同時に起動して同時に完了させる。

**スキルありの実行：**

```
このタスクを実行してください：
- スキルパス: <path-to-skill>
- タスク: <evalプロンプト>
- 入力ファイル: <evalファイルがあれば記載、なければ"none">
- 出力先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが求めるもの。例："the .docxファイル"、"最終CSV">
```

**ベースライン実行**（同じプロンプトで、ベースラインの内容はコンテキストによる）：
- **新規スキル作成時**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/` に保存。
- **既存スキル改善時**: 旧バージョン。編集前にスキルのスナップショットを取る（`cp -r <skill-path> <workspace>/skill-snapshot/`）し、ベースラインサブエージェントをそのスナップショットに向ける。`old_skill/outputs/` に保存。

各テストケースに `eval_metadata.json` を作成する（アサーションは今は空でよい）。evalのディレクトリ名はテストしている内容を表す説明的な名前にする（「eval-0」だけにしない）。このイテレーションで新しいまたは修正されたevalプロンプトを使う場合は、各新しいevalディレクトリにこのファイルを作成する。

```json
{
  "eval_id": 0,
  "eval_name": "説明的な名前",
  "prompt": "ユーザーのタスクプロンプト",
  "assertions": []
}
```

### ステップ2: 実行中にアサーションのドラフトを書く

実行完了を待つだけでなく、この時間を有効活用する。各テストケースの定量的アサーションをドラフトし、ユーザーに説明する。`evals/evals.json` に既にアサーションがある場合はレビューして内容を説明する。

良いアサーションは客観的に検証可能で、説明的な名前を持つ。ベンチマークビューアーで見たとき、各アサーションが何を確認しているか一目でわかるように。主観的なスキル（文体、デザイン品質）は定性的に評価する方が良い。

ドラフトができたら `eval_metadata.json` と `evals/evals.json` を更新する。ユーザーがビューアーで見るものも説明する（定性的な出力と定量的なベンチマークの両方）。

### ステップ3: 実行完了時にタイミングデータを記録する

各サブエージェントのタスクが完了すると、`total_tokens` と `duration_ms` を含む通知が来る。すぐに実行ディレクトリの `timing.json` に保存する：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

このデータを取得できる唯一のタイミング——タスク通知から来るもので他には残らない。まとめて処理しようとせず、通知が来たらすぐに処理すること。

### ステップ4: グレーディング、集計、ビューアーの起動

全実行が完了したら：

1. **各実行をグレーディング** — `agents/grader.md` を読んでアサーションを出力に照合するグレーダーサブエージェントを起動する（またはインラインでグレーディング）。結果を各実行ディレクトリの `grading.json` に保存する。grading.jsonのexpectations配列は `text`、`passed`、`evidence` フィールドを使うこと（`name`/`met`/`details` などの別の名前は不可）——ビューアーがこの正確なフィールド名に依存している。プログラムで確認できるアサーションはスクリプトを書いて実行する。目視よりも速く、信頼性が高く、イテレーションをまたいで再利用できる。

2. **ベンチマークへの集計** — skill-creatorディレクトリから集計スクリプトを実行：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   pass_rate、time、tokensを平均±標準偏差とデルタで示した `benchmark.json` と `benchmark.md` が生成される。benchmark.jsonを手動生成する場合はビューアーが期待する正確なスキーマを `references/schemas.md` で確認すること。
   with_skillバージョンをベースラインより前に配置すること。

3. **アナリストパス** — ベンチマークデータを読み、集計統計が隠しがちなパターンを見つける。`agents/analyzer.md`（「Analyzing Benchmark Results」セクション）を参照——スキルに関係なく常にパスするアサーション（識別力なし）、分散が高いeval（不安定な可能性）、時間・トークンのトレードオフなどを探す。

4. **ビューアーを起動する**（定性的な出力と定量的データの両方）：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降は `--previous-workspace <workspace>/iteration-<N-1>` も渡す。

   **Cowork / ヘッドレス環境:** `webbrowser.open()` が使えない場合やディスプレイがない場合は、サーバーを起動する代わりに `--static <output_path>` でスタンドアロンHTMLファイルを生成する。「Submit All Reviews」ボタンクリック時に `feedback.json` としてダウンロードされる。ダウンロード後、次のイテレーションで使うためにワークスペースディレクトリにコピーする。

注意：ビューアーはgenerate_review.pyを使って生成すること。カスタムHTMLを書く必要はない。

5. **ユーザーに伝える**：「ブラウザで結果を開きました。2つのタブがあります——『Outputs』では各テストケースを確認してフィードバックを書けます、『Benchmark』では定量的な比較が見られます。終わったら戻って教えてください。」

### ビューアーでユーザーが見るもの

「Outputs」タブは1テストケースずつ表示：
- **プロンプト**: 与えたタスク
- **出力**: スキルが生成したファイル（可能な限りインラインでレンダリング）
- **前回の出力**（イテレーション2以降）: 前のイテレーションの出力を折りたたんで表示
- **Formal Grades**（グレーディング実行済みの場合）: アサーションのpass/failを折りたたんで表示
- **フィードバック**: 入力しながら自動保存されるテキストボックス
- **前回のフィードバック**（イテレーション2以降）: 前回のコメントをテキストボックスの下に表示

「Benchmark」タブは統計サマリーを表示：各設定のpass rate、timing、token使用量、eval別の内訳、アナリストの観察。

ナビゲーションは前後ボタンまたは矢印キーで行う。終了したら「Submit All Reviews」をクリックすると全フィードバックが `feedback.json` に保存される。

### ステップ5: フィードバックを読む

ユーザーが完了を伝えたら `feedback.json` を読む：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "グラフに軸ラベルがない", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完璧、これがいい", "timestamp": "..."}
  ],
  "status": "complete"
}
```

フィードバックが空なら問題なしということ。ユーザーが具体的な不満を持つテストケースに集中して改善する。

ビューアーサーバーを停止する：

```bash
kill $VIEWER_PID 2>/dev/null
```

## スキルの改善

これがループの核心。テストケースを実行し、ユーザーがレビューした。次はフィードバックをもとにスキルを改善する。

### 改善の考え方

1. **フィードバックを一般化する。** ここでやっていることは、何百万回も使われ得るスキルを作ること。ユーザーと一緒に少数の例を反復しているのは速く動くためだ。でも、その例にしか機能しないスキルは無意味。細かく過適合した変更や厳しすぎる制約を加えるのではなく、頑固な問題があれば別のメタファーや作業パターンを試してみる。安上がりに試せて、うまくいくかもしれない。

2. **プロンプトを簡潔に保つ。** 役に立っていないものは削除する。最終出力だけでなくトランスクリプトを読むこと——スキルがモデルに無駄な作業をさせているようなら、そのスキルの該当部分を削除してどうなるか試す。

3. **理由を説明する。** モデルに求めることすべてについて「なぜ」を説明する。今のLLMは**賢い**。セオリー・オブ・マインドがあり、良いハーネスがあれば指示を超えて本当に成果を出せる。ユーザーのフィードバックが短くても、なぜそう書いたのかを実際に理解し、その理解を指示に反映させる。ALWAYSやNEVERを全大文字で書いたり、超厳格な構造を使おうとしたら黄色信号——可能なら、なぜそれが重要かを説明する形に書き換える。より人道的で、力強く、効果的なアプローチだ。

4. **テストケースをまたぐ繰り返し作業を探す。** テスト実行のトランスクリプトを読み、複数のサブエージェントが独立して同じようなヘルパースクリプトを書いたり同じ手順を踏んでいないか確認する。3つのテストケース全部で `create_docx.py` や `build_chart.py` が書かれていたら、スキルにそのスクリプトをバンドルする強いシグナル。一度書いて `scripts/` に入れ、スキルに使うよう指示する。将来の全ての実行で車輪の再発明を防げる。

このタスクはかなり重要。思考時間は惜しまず、しっかり考える。ドラフトを書いてから新鮮な目で見直し改善することをお勧めする。ユーザーの頭の中に入り込んで、何が欲しいのかを理解しようとすること。

### イテレーションループ

スキルを改善した後：

1. スキルに改善を適用する
2. 全テストケースを新しい `iteration-<N+1>/` ディレクトリで再実行する（ベースライン実行も含む）。新規スキル作成の場合、ベースラインは常に `without_skill`（スキルなし）——イテレーションをまたいで変わらない。既存スキルの改善の場合は、何がベースラインとして適切か判断する：ユーザーが持ち込んだ元のバージョンか、前のイテレーションか。
3. `--previous-workspace` で前のイテレーションを指定してレビュアーを起動する
4. ユーザーがレビューして完了を伝えるまで待つ
5. 新しいフィードバックを読み、改善して繰り返す

以下のいずれかになるまで続ける：
- ユーザーが満足したと言う
- フィードバックが全部空（全部OKということ）
- 意味のある改善が見込めなくなった

## 上級：ブラインド比較

2つのバージョンを厳密に比較したい場合（「新バージョンは本当に良くなった？」など）、ブラインド比較システムがある。詳細は `agents/comparator.md` と `agents/analyzer.md` を参照。基本的なアイデア：どちらが何かを教えずに2つの出力を独立したエージェントに渡して品質を判定させる。そして勝者がなぜ勝ったかを分析する。

任意の機能であり、サブエージェントが必要。ほとんどのユーザーには不要。人間レビューのループで十分。

## Descriptionの最適化

SKILL.mdフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決める主要な仕組み。スキルの作成・改善後は、トリガー精度を上げるためにdescriptionの最適化を提案する。

### ステップ1: トリガーevalクエリの生成

20個のevalクエリを作る——トリガーすべきものとすべきでないものを混ぜる。JSONで保存：

```json
[
  {"query": "ユーザーのプロンプト", "should_trigger": true},
  {"query": "別のプロンプト", "should_trigger": false}
]
```

クエリはClaude CodeまたはClaude.aiユーザーが実際に入力するようなリアルなもの。抽象的なリクエストではなく、具体的で詳細なもの。ファイルパス、仕事や状況の個人的な背景、列名と値、会社名、URL、少しの背景情報など。一部は小文字、略語、誤字、カジュアルな言葉遣いでもよい。長さはさまざまで、明確なものよりエッジケースに集中する。

NG例：`"データを整形して"`、`"PDFからテキストを抽出して"`、`"グラフを作って"`

OK例：`"上司からxlsxファイルが来たんですけど（ダウンロードに'Q4 sales final FINAL v2.xlsx'みたいな名前で入ってるはず）、利益率をパーセントで表す列を追加したいんです。収益はC列でコストはD列だったと思います"`

**should-trigger クエリ（8〜10個）**: カバレッジを考える。同じ意図の異なる言い方——フォーマルなものとカジュアルなものの両方。スキルの名前やファイルタイプを明示しなくても明らかに必要なケースも含める。珍しいユースケースや他のスキルと競合してもこのスキルが勝つべきケースも追加する。

**should-not-trigger クエリ（8〜10個）**: 最も価値があるのはニアミス——スキルとキーワードや概念を共有するが実際は別のものが必要なクエリ。隣接するドメイン、単純なキーワードマッチではトリガーしそうだがすべきでないあいまいな表現、スキルに関係することを含むが別のツールが適切なコンテキストなど。

重要な注意：should-not-triggerクエリを明らかに無関係なものにしない。「PDFスキルのネガティブテストとしてフィボナッチ関数を書いて」は簡単すぎて何も検証しない。ネガティブケースは本当にトリッキーなものにする。

### ステップ2: ユーザーとレビュー

HTMLテンプレートを使ってevalセットをユーザーに確認してもらう：

1. `assets/eval_review.html` からテンプレートを読む
2. プレースホルダーを置き換える：
   - `__EVAL_DATA_PLACEHOLDER__` → evalアイテムのJSON配列（クォートなし——JS変数の代入）
   - `__SKILL_NAME_PLACEHOLDER__` → スキルの名前
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → スキルの現在のdescription
3. tempファイルに書き出す（例：`/tmp/eval_review_<skill-name>.html`）して開く：`open /tmp/eval_review_<skill-name>.html`
4. ユーザーはクエリを編集、should-triggerをトグル、追加・削除して「Export Eval Set」をクリック
5. ファイルは `~/Downloads/eval_set.json` にダウンロードされる——複数ある場合（例：`eval_set (1).json`）は最新版を確認する

このステップは重要——evalクエリが悪いとdescriptionも悪くなる。

### ステップ3: 最適化ループの実行

ユーザーに伝える：「時間がかかります——バックグラウンドで最適化ループを実行して定期的に状況を報告します。」

evalセットをワークスペースに保存してからバックグラウンドで実行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

現在のセッションを動かしているモデルIDを使うこと（システムプロンプトから）。トリガーテストが実際のユーザー体験と一致する。

実行中は定期的に出力を確認して、現在のイテレーションとスコアの状況をユーザーに伝える。

完全な最適化ループを自動処理する。evalセットをtraining 60%とheld-out test 40%に分割し、現在のdescriptionを評価（各クエリを3回実行して信頼性の高いトリガー率を取得）、失敗したものに基づいてClaudeに改善案を提案させる。train/testの両方で各新しいdescriptionを再評価し、最大5回繰り返す。完了するとブラウザでイテレーション別の結果レポートを開き、`best_description` を含むJSONを返す——過学習を防ぐためtrainスコアではなくtestスコアで選択。

### スキルトリガーの仕組み

トリガーの仕組みを理解するとより良いevalクエリが書ける。スキルはClaudeの `available_skills` リストにname + descriptionとともに表示され、Claudeはそのdescriptionに基づいてスキルを参照するか決める。重要なポイント：Claudeは自分だけでは対処しにくいタスクにのみスキルを参照する——「このPDFを読んで」のような単純な一手順のクエリは、descriptionが完璧に一致していても基本ツールで直接対応できるためスキルをトリガーしないことがある。複雑で複数ステップや専門的なクエリは、descriptionが一致すれば確実にスキルをトリガーする。

evalクエリはClaudeがスキルを参照することで恩恵を受けるほど実質的なものにすること。「ファイルXを読んで」のような単純なクエリは質の悪いテストケース——descriptionの品質に関わらずスキルをトリガーしない。

### ステップ4: 結果の適用

JSONの `best_description` を取得してスキルのSKILL.mdフロントマターを更新する。before/afterをユーザーに見せてスコアを報告する。

### パッケージングと提出（`present_files` ツールが利用可能な場合のみ）

`present_files` ツールが利用可能か確認する。なければこのステップをスキップ。あればスキルをパッケージングして.skillファイルをユーザーに提示する：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

パッケージング後、生成された `.skill` ファイルのパスをユーザーに伝えてインストールできるようにする。

## Claude.ai固有の手順

Claude.aiのコアワークフローは同じ（ドラフト→テスト→レビュー→改善→繰り返し）だが、サブエージェントがないためいくつかの仕組みが変わる。以下の点を適応させること：

**テストケースの実行**: サブエージェントがないため並列実行不可。各テストケースで、スキルのSKILL.mdを読み、自分でその指示に従ってテストプロンプトを達成する。一度に一つずつ。独立したサブエージェントより厳密さは劣る（スキルを書いた自分が実行するので全コンテキストがある）が、有効な健全性チェックになる。ベースライン実行はスキップ——スキルを使って要求されたタスクを完了するだけでよい。

**結果のレビュー**: ブラウザが開けない場合（Claude.aiのVMにディスプレイがない、リモートサーバーなど）、ブラウザレビューアーを完全にスキップ。代わりに会話内で直接結果を提示する。各テストケースでプロンプトと出力を見せる。ユーザーが確認する必要があるファイル（.docx、.xlsxなど）はファイルシステムに保存してパスを伝えダウンロードして確認してもらう。インラインでフィードバックを求める：「どうですか？変えたいところはありますか？」

**ベンチマーク**: 定量ベンチマークはスキップ——サブエージェントなしではベースライン比較が意味をなさない。ユーザーからの定性的フィードバックに集中する。

**イテレーションループ**: 同じ——スキルを改善し、テストケースを再実行し、フィードバックを求める。ただし途中にブラウザレビューアーはない。ファイルシステムがあればイテレーションディレクトリに結果を整理してもよい。

**Description最適化**: `claude` CLIツール（具体的には `claude -p`）が必要で、Claude Codeでのみ利用可能。Claude.aiではスキップ。

**ブラインド比較**: サブエージェントが必要。スキップ。

**パッケージング**: `package_skill.py` はPythonとファイルシステムがあれば動く。Claude.aiでも実行でき、ユーザーは生成された `.skill` ファイルをダウンロードできる。

**既存スキルの更新**: ユーザーが新規作成ではなく既存スキルの更新を求めている場合：
- **元の名前を保持する。** スキルのディレクトリ名と `name` フロントマターフィールドを確認し、変えずに使う。例：インストール済みのスキルが `research-helper` なら `research-helper.skill` を出力する（`research-helper-v2` にしない）。
- **編集前に書き込み可能な場所にコピーする。** インストール済みのスキルパスは読み取り専用の場合がある。`/tmp/skill-name/` にコピーしてそこで編集し、そこからパッケージングする。
- **手動でパッケージングする場合は `/tmp/` でステージングしてから**出力ディレクトリにコピーする——直接書き込みは権限の問題で失敗することがある。

## Cowork固有の手順

Coworkを使っている場合の主なポイント：

- サブエージェントが使えるので、メインワークフロー（テストケースの並列起動、ベースライン実行、グレーディングなど）はすべて動く。（タイムアウトの問題が深刻な場合は直列実行でもOK。）
- ブラウザやディスプレイがないので、evalビューアーを生成するときはサーバーを起動する代わりに `--static <output_path>` でスタンドアロンHTMLファイルを生成する。ユーザーがクリックできるリンクを提示する。
- Coworkのセットアップではテスト実行後にClaudeがevalビューアーを生成しない傾向がある。念のため再強調：Claude CodeでもCoworkでも、テスト後は必ずevalビューアーを生成してから自分で評価・修正を行うこと。`generate_review.py` を使うこと（独自HTMLを書かない）。全大文字で書かなければならないが：evalビューアーを生成してから自分で評価すること。人間にできるだけ早く見せること！
- フィードバックの仕組みが異なる：サーバーが動いていないため、ビューアーの「Submit All Reviews」ボタンは `feedback.json` ファイルとしてダウンロードする。そこから読み込む（先にアクセス権のリクエストが必要な場合がある）。
- パッケージングは動く——`package_skill.py` はPythonとファイルシステムがあればよい。
- Description最適化（`run_loop.py` / `run_eval.py`）はブラウザではなくサブプロセスで `claude -p` を使うためCoworkでも問題なく動くはず。ただしスキルが完全に完成してユーザーが満足するまで後回しにすること。
- **既存スキルの更新**: ユーザーが既存スキルの更新を求めている場合はclaude.aiセクションの更新ガイダンスに従うこと。

## このユーザー固有のガイドライン

スキルのSKILL.mdを書く際は、以下の追加ルールにも従うこと。

1. **目的・トリガー条件を先頭に明記すること**
   - スキルファイルの冒頭に「いつ・どんな状況でこのスキルが使われるか」を書く
   - フロントマターの `description` にトリガーキーワードを含める

2. **手順はステップごとに番号付きで記載すること**
   - `## ステップ1:`, `## ステップ2:` のように番号と目的を明示する
   - 各ステップは独立して理解できる粒度にする

3. **各ステップに期待される出力を明記すること**
   - ステップの末尾に「期待される出力:」や「確認方法:」を書く
   - ツール呼び出しの結果がどうあるべきかを明示する

4. **完了後の確認・報告手順を含めること**
   - 最後のステップとして「完了条件」と「ユーザーへの報告内容」を書く
   - VOICEVOXで音声通知する場合はそのテキストも指定する

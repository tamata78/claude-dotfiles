# feature-dev スキル

## 目的・トリガー条件

`/feature-dev` または「feature-dev を実行して」で起動。

`$ARGUMENTS` に機能の説明・要件・チケット番号が渡される場合はそれを使う。渡されない場合はユーザーに概要と BTS チケット番号を確認してから進める。

全体の流れ:
要件整理 → **レビュー観点ロード** → 計画策定(Plan) → **サブエージェント批判レビュー** → 承認 → **ブランチ作成** → 実装 → ビルド → テスト → 報告 → スナップショット

---

## ステップ1: 要件整理とコードベース調査

`$ARGUMENTS` を解析し要件を整理する。

1. 関連ファイル・モジュールを Glob/Grep/Read で調査する（3クエリ以上の探索は `Explore` サブエージェントに委譲）
2. 既存パターン・規約を把握する（命名規則、ディレクトリ構成、レイヤー構成、API設計）
3. 影響範囲を特定する（変更・追加するファイル）
4. `wiki/dev/{プロジェクト}/` を参照し、設計が wiki の既存知見と矛盾しないか確認する。矛盾する場合は警告する

**出力:** 対象ファイルのリストと変更方針の要約

---

## ステップ2: レビュー観点の事前ロード（設計に反映するため）

設計を始める前に、対象リポジトリのレビュー観点を読み込み、**設計段階から規約・過去指摘を考慮する**。後工程の指摘を先回りして潰すのが狙い。

1. 共有フラグメント `~/.claude/shared/review-rules.md` を Read する
2. その手順（Step A）に従い、git remote から `PROJECT_KEY` を導出する
   （例: `demae-api-shiva` → `shiva`、`demae-api-odin` → `odin`）
3. `~/workspace/reviews/guidelines/` 配下を **プロジェクト別** に読み込む:
   - `review-bot-<key>.yml` — コーディング規約（例: shiva の一方通行原則 Controller→UseCase→Service→Repository、DB型マッピング、`!!` 禁止）
   - `pr-review-insights-common.md` — 汎用観点（Null安全 / エラー処理 / セキュリティ / パフォーマンス）
   - `pr-review-insights-<key>.md` — そのリポジトリの過去 PR 指摘事例
   - 見つからない場合は無印ファイルを fallback、それも無ければ共通観点のみで続行

**出力:** 設計時に守るべき規約・過去指摘のチェックリスト（後続ステップで参照）

---

## ステップ3: 実装計画の策定（Plan モード）

`EnterPlanMode` で実装計画を作成する。ステップ2のレビュー観点を**計画に織り込む**。

計画に含めること:
- 新規作成するファイル
- 変更するファイルと変更箇所
- 設計上の判断とその理由
- 考慮したが採用しなかった代替案
- **ステップ2のレビュー観点との整合**（守る規約・回避する過去指摘パターンを明記）
- テスト戦略

影響ファイル数・複雑度を見積もり、計画冒頭に小規模／大規模の別を記す。

**出力:** ファイル別の変更内容リストと設計説明（この時点ではまだユーザー承認しない）

---

## ステップ4: 設計の批判的レビュー（サブエージェント）

ステップ3の計画を、**別のサブエージェントに批判的レビューさせ**、その結果で計画を改善してからユーザーに出す。自分の前提に同意させず、具体的な改善提案を引き出す。

1. `Agent` ツールでサブエージェントを起動する（`subagent_type` は `Plan` を推奨。広範な再調査が要るなら `general-purpose`）
2. サブエージェントに渡す情報:
   - 策定した実装計画（全文）
   - ステップ2でロードしたレビュー観点（規約・過去指摘・wiki 知見）
   - 対象ファイルと現状コードの要点
3. サブエージェントへの指示（批判レビューの観点）:
   - 設計上の見落とし／レイヤリング違反／責務分離の問題
   - 過去 PR 指摘の再発リスク（`pr-review-insights-<key>.md` と照合）
   - エッジケース・エラー処理・並行性・パフォーマンス（N+1、不要ループ）の漏れ
   - よりシンプルな代替設計の有無
   - テスト戦略の網羅性・モック化の妥当性
   - **「妥当です」で終わらせず、必ず具体的な改善点か反証を返すこと**
4. 返ってきた指摘を精査し、計画に反映する。採否を判断し、**却下した指摘は理由を残す**
5. 改善後の計画を `ExitPlanMode` でユーザーに提示し、承認を得る

**出力:** 批判レビューの要点と、それを受けた計画の変更点

---

## ステップ5: feature ブランチ作成

実装に着手する前に、専用の feature ブランチを作成する。**ブランチ作成元（ベースブランチ）を必ず確認する。**

1. **ベースブランチを検出してユーザーに確認する**
   - デフォルトブランチを検出:
     `git symbolic-ref refs/remotes/origin/HEAD | sed 's@refs/remotes/origin/@@'`
     （shiva/odin → `master`、order-service → `develop`、settlement-service → `main` などリポジトリで異なる）
   - 特定リリースブランチから派生させたい場合もあるため、検出結果を提示して確認する
   - 確定後に最新化: `git fetch origin <base>` → `git switch <base>` → `git pull --ff-only`

2. **ブランチ名を組み立てる** — 形式: `feature/<BTSチケット番号>-<修正概要(英字 kebab-case)>`
   - 例: `feature/DEMAE-48586-shorten-bulk-cancel-time`
   - BTS チケット番号は `$ARGUMENTS` から抽出。無ければユーザーに確認
   - 修正概要は実装内容から英語で簡潔に（小文字 kebab-case、3〜5語）
   - **区切り文字の慣習はリポジトリに合わせる**。直近の feature ブランチを確認:
     `git branch -a --sort=-committerdate | grep -E 'feature/(DEMAE|NOISSUE)' | head -5`
     - チケット番号の後ろが `_` のリポジトリ（例: shiva `feature/DEMAE-47329_switch_coupon_check_api_v2`）→ `_` 区切り
     - `-` のリポジトリ（例: odin `feature/DEMAE-48735-softbank-error-handling`）→ `-` 区切り
   - チケットが無い作業は既存慣習に従い `NOISSUE-<概要>` を使う

3. **組み立てたブランチ名をユーザーに提示して確認**してから作成:
   `git switch -c feature/DEMAE-XXXXX-<概要>`

**出力:** 作成したブランチ名とベースブランチ

---

## ステップ6: 実装

承認された計画に従ってコードを実装する。

- 既存のコードスタイル・規約に合わせる（ステップ2の規約を遵守）
- 不要なリファクタリングや追加機能は入れない（最小限の変更）
- セキュリティ上の問題（XSS、各種インジェクション、秘匿情報のログ出力）を作り込まない

**出力:** 変更・作成したファイル一覧

---

## ステップ7: ビルド確認

プロジェクトのビルドコマンドを実行しエラーが無いことを確認する。

- Kotlin/Gradle: `./gradlew build` または `./gradlew compileKotlin`、フォーマット `./gradlew ktlintFormat`、静的解析 `./gradlew detekt`
- TypeScript: `npx tsc --noEmit` または `npm run build`
- エラーがあれば即修正してから次へ

**確認方法:** ビルドコマンドの exit code が 0

---

## ステップ8: テスト確認

テストが存在する場合は実行し全テストがパスすることを確認する。

- 例: `./gradlew test`, `npm test`, `pytest`
- 失敗した場合は修正してから次へ
- テストが存在しない場合はその旨をユーザーに伝える

**確認方法:** テストコマンドの exit code が 0

---

## ステップ9: 完了報告

実装内容を報告する:
- 実装した機能の概要
- 作成したブランチ名・ベースブランチ
- 変更・作成したファイルの一覧
- ビルド・テストの結果
- 批判レビューで指摘され反映／却下した点
- 手動で確認すべき動作（あれば）

VOICEVOX で音声通知:
```
vv "実装完了なのだ！確認してほしいのだ"
```

---

## ステップ10: 開発スナップショットの更新

`project-snapshot` スキルで今回の実装知見を記録する:
- Plan 設計の判断・採用しなかった代替案
- サブエージェント批判レビューの結論
- 実装中に気づいたこと・注意点
- 変更したファイルとその役割

次回セッションで即座に開発文脈を把握できる状態を保つ。

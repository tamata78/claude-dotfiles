# feature-dev スキル

## 目的・トリガー条件

このスキルは `/feature-dev` または「feature-dev を実行して」と言われたときに使用する。

引数（`$ARGUMENTS`）として機能の説明や要件が渡される場合はそれを使い、渡されない場合はユーザーに機能の概要を確認してから進める。

---

## ステップ1: 要件整理とコードベース調査

`$ARGUMENTS` を解析し、実装する機能の要件を整理する。

1. 関連するファイル・モジュールを Glob/Grep/Read で調査する
2. 既存のパターン・規約を把握する（命名規則、ディレクトリ構成、API設計など）
3. 影響範囲を特定する（どのファイルを変更・追加するか）

**期待される出力:** 対象ファイルのリストと変更方針の要約

---

## ステップ2: 実装計画の策定（Plan モード）

`EnterPlanMode` を使って実装計画を作成し、ユーザーに確認を取る。

計画には以下を含めること：
- 新規作成するファイル
- 変更するファイルと変更箇所
- 設計上の判断とその理由
- 考慮したが採用しなかった代替案

ユーザーが計画を承認したら `ExitPlanMode` して実装に進む。

**期待される出力:** ファイル別の変更内容リストと設計説明

---

## ステップ3: 実装

計画に従ってコードを実装する。

- 既存のコードスタイル・規約に合わせる
- 不要なリファクタリングや追加機能は入れない（最小限の変更）
- セキュリティ上の問題（XSS、インジェクションなど）が発生しないよう注意する

**期待される出力:** 変更・作成したファイル一覧

---

## ステップ4: ビルド確認

プロジェクトのビルドコマンドを実行してエラーがないことを確認する。

- TypeScript プロジェクトなら `npx tsc --noEmit` または `npm run build`
- エラーがあれば即座に修正してから次へ進む

**確認方法:** ビルドコマンドの exit code が 0 であること

---

## ステップ5: テスト確認

テストが存在する場合はテストコマンドを実行して全テストがパスすることを確認する。

- テストコマンドの例: `npm test`, `npm run test`, `pytest` など
- テストが失敗した場合は修正してから次へ進む
- テストが存在しない場合はその旨をユーザーに伝える

**確認方法:** テストコマンドの exit code が 0 であること

---

## ステップ6: 完了報告

実装内容をユーザーに報告する。

報告内容：
- 実装した機能の概要
- 変更・作成したファイルの一覧
- ビルド・テストの結果
- 手動で確認すべき動作（あれば）

VOICEVOXで音声通知：
```
vv "実装完了なのだ！確認してほしいのだ"
```

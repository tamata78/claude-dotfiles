# feature-dev スキル

## 目的・トリガー条件

このスキルは `/feature-dev` または「feature-dev を実行して」と言われたときに使用する。

引数（`$ARGUMENTS`）として機能の説明や要件が渡される場合はそれを使い、渡されない場合はユーザーに機能の概要を確認してから進める。

---

## ステップ1: 要件整理とコードベース調査

`$ARGUMENTS` を解析し、実装する機能の要件を整理する。

1. 関連するファイル・モジュールを Glob/Grep/Read で調査する
2. 既存のパターン・規約を把握する（命名規則、ディレクトリ構成、API設計など）
3. 影響範囲を特定する（どのファイルを変更・追加するか）

**期待される出力:** 対象ファイルのリストと変更方針の要約

---

## ステップ2: 実装規模の判定

影響ファイル数・複雑度を判定する:
- **小規模**（5ファイル未満・単一機能）: 通常の実装フローで進める
- **大規模**（5ファイル以上・複数モジュール横断・アーキテクチャ変更）: ステップ2cでPlanレビュアーを自動起動

大規模の場合はステップ2b（Plan策定）後に **ステップ2c（Planレビュアー自動起動）** を実行する。

## ステップ2b: 実装計画の策定（Plan モード）

`EnterPlanMode` を使って実装計画を作成し、ユーザーに確認を取る。

計画には以下を含めること：
- 新規作成するファイル
- 変更するファイルと変更箇所
- 設計上の判断とその理由
- 考慮したが採用しなかった代替案

ステップ2で「大規模」と判定された場合、ユーザーへの提示の前にステップ2cでレビュアーを起動してPlanを精査する。ユーザーが計画を承認したら `ExitPlanMode` して実装に進む。

**期待される出力:** ファイル別の変更内容リストと設計説明

---

## ステップ2c: Planレビュアー並走（大規模時のみ）

**起動条件**: ステップ2で「大規模」（5ファイル以上 or 複数モジュール横断）と判定された場合のみ実行。小規模はスキップしてステップ3へ。
- `$ARGUMENTS` に `--review` があれば小規模でも強制起動
- `$ARGUMENTS` に `--no-review` があれば大規模でもスキップ
- ユーザーが「レビュー不要」と明示した場合もスキップ

### レビュアーサブエージェントの起動

`Agent` ツールで以下の設定で起動する:

```
subagent_type: general-purpose
description: "Plan review (independent)"
```

プロンプトは以下のテンプレートに実際の値を埋め込んで渡す:

---
あなたは実装者と独立したレビュアーです。これから渡す実装Planを、本プロジェクトのレビュー規約に照らして批評してください。

**【規約読み込み】**
1. `/Users/tamata78/.claude/shared/review-rules.md` を Read し Step A〜D に従う
2. Step A のプロジェクトキー判定 / review-bot / insights ファイル探索を実行

**【レビュー対象】**
- ユーザー要件: {{要件サマリ}}
- 影響ファイル候補: {{ファイルリスト}}
- 設計上の判断: {{判断と理由}}
- 採用しなかった代替案: {{代替案}}
- 実装Plan本文:
{{Plan全文}}

**【観点】**
review-rules.md Step B（B.1〜B.4）に加えて以下を重点確認:
- 設計の妥当性: 責務分離 / レイヤリング / 一方通行原則
- 影響範囲の漏れ: Planに含まれていないが触るべきファイル / 既存パターンとの整合性
- エッジケース: NULL安全 / 失敗系 / 並行性 / 既存データ移行
- UIコード観点（React/Tailwind系の変更がある場合）: コンポーネント責務 / hooks / アクセシビリティ / 状態管理 / 既存コンポーネント再利用の余地
- テスト方針: 何をどう検証するか書かれているか

**【出力フォーマット】**
```markdown
## Plan Review Summary
総評（1-2行）

## 🔴 Critical（修正必須）
（指摘 / 理由 / 修正案。なければ「指摘なし」）

## 🟡 Warning（強く推奨）
（同上）

## 💡 Suggestion（軽微）
（同上）

## ✅ 良い点
（任意）
```
コードを書かない、ファイルを編集しない、読み取りと分析のみ。400〜800字目安。
---

### レビュー結果の反映

1. レビュアーの返答を受信する
2. **🔴 Critical** は Plan に必ず反映する
3. **🟡 Warning** は採用・不採用の判断と理由を明示する
4. **💡 Suggestion** は任意で取り込む
5. 修正後の Plan差分（採用した指摘・採用しなかった理由）をユーザーに提示する
6. ユーザーが承認したら `ExitPlanMode` → ステップ3 実装へ

---

## ステップ3: 実装

計画に従ってコードを実装する。

- 既存のコードスタイル・規約に合わせる
- 不要なリファクタリングや追加機能は入れない（最小限の変更）
- セキュリティ上の問題（XSS、インジェクションなど）が発生しないよう注意する

**期待される出力:** 変更・作成したファイル一覧

---

## ステップ4: ビルド確認

プロジェクトのビルドコマンドを実行してエラーがないことを確認する。

- TypeScript プロジェクトなら `npx tsc --noEmit` または `npm run build`
- エラーがあれば即座に修正してから次へ進む

**確認方法:** ビルドコマンドの exit code が 0 であること

---

## ステップ5: テスト確認

テストが存在する場合はテストコマンドを実行して全テストがパスすることを確認する。

- テストコマンドの例: `npm test`, `npm run test`, `pytest` など
- テストが失敗した場合は修正してから次へ進む
- テストが存在しない場合はその旨をユーザーに伝える

**確認方法:** テストコマンドの exit code が 0 であること

---

## ステップ6: 完了報告

実装内容をユーザーに報告する。

報告内容：
- 実装した機能の概要
- 変更・作成したファイルの一覧
- ビルド・テストの結果
- 手動で確認すべき動作（あれば）

VOICEVOXで音声通知：
```
vv "実装完了なのだ！確認してほしいのだ"
```

---

## ステップ7: 開発スナップショットの更新

`project-snapshot` スキルを使って、今回の実装知見をスナップショットに記録する。

- 今回のPlan設計の判断・採用しなかった代替案
- 実装中に気づいたこと・注意点
- 変更したファイルとその役割

これにより、次回セッションで即座に開発文脈を把握できる状態を保つ。
